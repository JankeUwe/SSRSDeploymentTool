Abschnitt 1

1Überblick

Das SSRS Report Deployment Tool v3 ist ein WinForms-basiertes PowerShell-Skript für die standardisierte Verteilung von SSRS-Artefakten auf SQL Server Reporting Services 2019, 2022 und 2025. Die Kommunikation erfolgt ausschließlich über die REST API v2.0.

Funktionsumfang

  • WinForms-GUI mit geteiltem Layout (TreeView | Dateiliste | Log)
  • Automatische Erkennung eines lokalen SSRS-Servers beim Start
  • Windows-Authentifizierung oder manuelle Credentials (DOMAIN\User + Passwort)
  • Rekursive Ordneranlage im Report Server per REST
  • Datei-Scan des Quellverzeichnisses mit Checkbox-Auswahl
  • Farbcodiertes Deployment-Log (RichTextBox) mit Export-Funktion
  • Thread-sicheres GUI-Update per MethodInvoker
  • Fortschrittsanzeige (ProgressBar) während des Deployments

3REST API Kommunikation

3.1 API-Basis-URL (Get-SSRSApiBase)

Eingabe-URLNormalisiert auf
http://server/Reportshttp://server/Reports/api/v2.0
http://server/ReportServerhttp://server/Reports/api/v2.0
http://server/Reports/api/v2.0unveränd

3.2 Invoke-SSRSRequest

Zentrale REST-Hilfsfunktion für alle HTTP-Aufrufe. Unterstützt:

  • GET, POST, PATCH-Methoden
  • Windows-Authentifizierung (UseDefaultCredentials) oder PSCredential
  • Automatische JSON-Serialisierung (ConvertTo-Json -Depth 10)
  • Detailliertes Fehlerhandling: Response-Body, gesendeter Body und URI im Fehlertext

3.3 Ordner-Operationen

FunktionMethodeEndpunktVerhalten
Get-SSRSFoldersGET/Folders?$orderby=PathAlle Ordner sortiert nach Pfad
Get-SSRSFolderExistsGET/Folders('<path>')true/false – kein Fehler bei 404
New-SSRSFolderSinglePOST/FoldersLegt Ordner an; HTTP 409 wird ignoriert
New-SSRSFolderRecursiveinternrekursivLegt alle fehlenden Elternordner an

3.4 Deploy-Operationen

FunktionNeu (POST)VorhandenContent Base64
Deploy-ReportPOST /ReportsPATCH /Reports('path')Ja (.rdl)
Deploy-DataSourcePOST /DataSourcesSKIP – keine ÜberschreibungNein (XML-Parsing)
Deploy-SharedDatasetPOST /DataSetsPATCH /DataSets('path')Ja (.rsd)
Besonderheit Datenquellen: Bestehende Datenquellen-Konfigurationen (Connection String, Credentials) bleiben beim Deployment immer erhalten. ConnectionString und Extension werden aus dem XML der .rds-Datei ausgelesen.
Abschnitt 4

4Automatische Server-Erkennung

Beim Start prüft Find-LocalSSRS vier URL-Kandidaten in dieser Reihenfolge. Die erste erreichbare URL wird im Eingabefeld vorbelegt. Jeder Kandidat wird mit TimeoutSec=4 und Windows-Auth gegen /api/v2.0/System geprüft.

#URL-Kandidat
1http://<COMPUTERNAME>/Reports
2http://<COMPUTERNAME>/ReportServer
3http://localhost/Reports
4http://localhost/ReportServer

Die Funktion gibt $null zurück, wenn kein Server gefunden wurde. In diesem Fall erscheint eine Warnung im Log und das Eingabefeld bleibt leer.

Abschnitt 5

5GUI-Threading

Lange REST-Operationen werden durch direkten Aufruf von Application.DoEvents() im Deploy-Loop entkoppelt. Alle GUI-Zugriffe folgen festen Sicherheitsregeln.

RegelUmsetzung
GUI-Werte als SnapshotAlle Control-Werte vor dem Loop lesen – nie im Loop direkt zugreifen
GUI-Updates per Invoke$control.Invoke([MethodInvoker]{...}) – keine direkte Property-Zuweisung
DoEvents im Deploy-LoopApplication.DoEvents() nach jedem Datei-Deployment
Buttons deaktivierenDeploy, Scan, Verbinden werden gesperrt während des Deployments

Thread-sicheres Log-Schreiben

PowerShell 5.1 
function Write-Log {
    param([string]$Msg, [string]$Lv='Info')
    $col = switch($Lv) {
        'Success' { $cOk }   'Warning' { $cWarn }
        'Error'   { $cErr }  'Skip'    { $cSkip }
        'Header'  { $cAccent }  default   { $cLogFg }
    }
    $rtb.SelectionColor = $col
    $rtb.AppendText("[$(Get-Date -Format 'HH:mm:ss')] $Msg`n")
    $rtb.ScrollToCaret()
}
Wichtig: In PowerShell 5.1 muss [System.Windows.Forms.MethodInvoker] verwendet werden. [System.Action] wird als Cmdlet-Name interpretiert und führt zu einem CommandNotFoundException.

Abschnitt 7

7Deployment-Ablauf

7.1 Startsequenz

  1. Form initialisieren, SplitterDistance setzen
  2. Find-LocalSSRS: lokalen Report Server suchen und URL vorbelegen
  3. Hinweismeldungen im Log ausgeben (Benutzer, Workflow-Schritte 1–5)

7.2 Verbindungsaufbau

  1. URL normalisieren via Get-SSRSApiBase
  2. Optionale PSCredential aufbauen (bei manuellen Credentials)
  3. Test-SSRSConnection: GET /System – liefert ProductName und ProductVersion
  4. Bei Erfolg: Load-Tree – Ordnerstruktur laden und TreeView aufbauen
  5. Update-DeployBtn: Deployment-Button freischalten wenn alle Voraussetzungen erfüllt

7.3 Deployment-Schleife

  1. Zielordner validieren und ggf. rekursiv anlegen (New-SSRSFolderRecursive)
  2. Alle angehakten ListViewItems durchlaufen
  3. Je nach Typ Deploy-Report / Deploy-DataSource / Deploy-SharedDataset aufrufen
  4. Ergebnis (CREATED / UPDATED / SKIPPED / FEHLER) im ListView und Log ausgeben
  5. ProgressBar erhöhen, Application.DoEvents() aufrufen
  6. Abschlussmeldung mit Zähler OK / Übersprungen / Fehler ausgeben
  7. Ordnerstruktur neu laden (Load-Tree) und Buttons reaktivieren

Ausführung

PowerShell 
powershell.exe -ExecutionPolicy Bypass -File ".\SSRS_DeployTool_v3.ps1"

Einschränkungen

  • Keine Unterstützung von SSRS-Subscriptions oder Berechtigungs-Deployments
  • Unterverzeichnisse im Quellpfad werden nicht rekursiv gescannt
  • Kerberos-Delegation (Double-Hop) nicht unterstützt – bei Remote-Servern manuelle Credentials verwenden
  • Schließen des Fensters während eines laufenden Deployments kann zu inkonsistenten Zuständen führen
Abschnitt 10

10Erweiterungshinweise

Neue Datenquellentypen

Deploy-DataSource liest Extension und ConnectString aus dem .rds XML. Für ODBC- oder Oracle-Quellen muss der DataSourceType-Wert entsprechend angepasst werden.

Rekursiver Verzeichnis-Scan

Der aktuelle Scan-Button ruft Get-ChildItem ohne -Recurse auf. Für rekursives Scanning genügt das Flag; der Zielordner im Report Server wird analog zur Verzeichnisstruktur über New-SSRSFolderRecursive aufgebaut.

HTTPS / Selbstsignierte Zertifikate

PowerShell 
# Vor dem ersten Invoke-RestMethod einfügen:
[System.Net.ServicePointManager]::ServerCertificateValidationCallback = { $true }
[System.Net.ServicePointManager]::SecurityProtocol = 'Tls12'

Persistentes Logging

Write-Log schreibt ausschließlich in die RichTextBox. Für persistentes Logging genügt es, parallel per Add-Content in eine Logdatei zu schreiben. Der Log speichern-Button exportiert den gesamten RichTextBox-Inhalt als UTF-8-Textdatei.
