Abschnitt 01

1Überblick

Das SSRS Deployment Tool automatisiert den Rollout von Report-Dateien auf SQL Server Reporting Services (SSRS) und Power BI Report Server (PBIRS). Es ersetzt fehleranfällige manuelle Uploads über den Report Manager durch einen standardisierten, protokollierten Deployment-Prozess per WinForms-GUI.

Das Tool kommuniziert direkt mit der SSRS REST API v2.0 — demselben Endpunkt, den sowohl klassische SSRS-Server als auch PBIRS exponieren. Für .pbix-Dateien wird ein eigener Multipart-Upload-Endpunkt verwendet, der nur auf PBIRS-Instanzen verfügbar ist.

Prinzip: Wird ausgebaut, nicht neu geschrieben — alle v3-Funktionen (Deploy-Report, Deploy-DataSource, Deploy-SharedDataset, Folder-Browsing) bleiben unverändert. Neue Funktionalität wird additiv hinzugefügt.

3Unterstützte Datei-Typen

Erweiterung Typ SSRS (klassisch) Power BI Report Server API-Endpunkt
.rdl Paginated Report ✓ ✓ POST /Reports
.rds Report Data Source ✓ ✓ POST /DataSources
.rsds Report Data Source (Shared) ✓ ✓ POST /DataSources
.rsd Shared Dataset ✓ ✓ POST /DataSets
.pbix NEU Power BI Desktop Report ausgegraut ★ aktiv POST /PowerBIReports
Hinweis: .pbix-Steuerelemente sind bei Verbindung zu einem reinen SSRS-Server sichtbar, aber deaktiviert (ausgegraut). Es erscheint kein Fehler — .pbix-Einträge werden mit Status SKIP protokolliert. Sobald ein PBIRS erkannt wird, werden die Steuerelemente automatisch aktiviert.
Abschnitt 04

4PBIRS-Erkennung & .pbix DeployNEU

Beim Verbinden mit einem Server ruft das Tool automatisch den System-Endpunkt ab und prüft das Feld ProductName. Enthält es „Power BI“, wird der Server als PBIRS klassifiziert und das Tool schaltet .pbix-Support frei.

REST · PBIRS-Erkennung 
# Schritt 1: System-Info abrufen
GET /Reports/api/v2.0/System

# Antwort bei klassischem SSRS:
{ "ProductName": "Microsoft SQL Server Reporting Services", ... }

# Antwort bei Power BI Report Server:
{ "ProductName": "Microsoft Power BI Report Server", ... }

# Schritt 2: .pbix deployen (nur wenn PBIRS erkannt)
POST /Reports/api/v2.0/PowerBIReports
Content-Type: multipart/form-data
Body: file=<pbix-Binaerinhalt>

Implementierung in PowerShell 5.1

Der Multipart-Upload nutzt System.Net.Http.HttpClient und MultipartFormDataContent — beide in PowerShell 5.1 über Add-Type -AssemblyName System.Net.Http verfügbar. Synchroner Aufruf via .GetAwaiter().GetResult().

PowerShell 5.1 · Deploy-PowerBIReport (Kernlogik) 
Add-Type -AssemblyName System.Net.Http

$handler = New-Object System.Net.Http.HttpClientHandler
$handler.UseDefaultCredentials = $true          # Windows Auth
$client  = New-Object System.Net.Http.HttpClient($handler)

$bytes = [System.IO.File]::ReadAllBytes($FilePath)
$mp    = New-Object System.Net.Http.MultipartFormDataContent
$fc    = New-Object System.Net.Http.ByteArrayContent(,$bytes)
$fc.Headers.ContentType = [System.Net.Http.Headers.MediaTypeHeaderValue]::Parse('application/octet-stream')
$mp.Add($fc, 'file', [System.IO.Path]::GetFileName($FilePath))

$resp = $client.PostAsync("$ApiBase/PowerBIReports", $mp).GetAwaiter().GetResult()
Fehler 403 (Lizenz): Wenn PBIRS ohne gültige Power BI-Lizenz betrieben wird, antwortet der Server mit HTTP 403. Das Tool fängt diesen Fehler ab und zeigt eine verständliche Meldung — kein roher Stack-Trace.

Schritt-für-Schritt

  1. Verbinden — Quell-Server und Ziel-Server getrennt verbinden (Windows Auth oder manuelle Credentials)
  2. Ordnerstruktur laden — TreeViews (links: Quelle, rechts: Ziel) werden befüllt
  3. Exportpfad wählen — lokales Verzeichnis für das Zwischenergebnis
  4. Connection String Mapping konfigurieren (optional) — beliebig viele Suchen/Ersetzen-Regeln im DataGridView
  5. Migrieren — das Tool exportiert alle Reports, Datenquellen und Datasets vom Quell-Server ins lokale Verzeichnis
  6. Zwischenverzeichnis prüfen (optional) — Dateien können vor dem Deploy inspiziert, geändert oder bei Bedarf manuell korrigiert werden
  7. Deploy — alle exportierten Dateien werden auf den Ziel-Server deployed; Ordnerstruktur wird automatisch angelegt
  8. Log-Auswertung — jede Datei wird mit Status CREATED / UPDATED / ERROR protokolliert

Was exportiert wird

TypQuelleExport-MethodeDatei

Warum Zwischenverzeichnis? Kein direkter Server-zu-Server-Transfer — alle exportierten Dateien liegen zuerst lokal. Das ermöglicht Inspektion, manuelle Korrekturen und bei Bedarf einen vollständigen Rollback ohne Datenverlust.
Abschnitt 08

8Connection String MappingNEU

Quell- und Ziel-Server haben in aller Regel unterschiedliche Datenbank-Server-Namen. Das Connection String Mapping löst dieses Problem: im DataGridView werden beliebig viele Suchen/Ersetzen-Regeln definiert, die während des Exports auf alle .rds-Datenquellen-Dateien angewendet werden.

#Suchen (Regex)Ersetzen durchAnwendung
1ssqlprod17\\INST01pbirs-server\INST01Server-Rename
2Data Source=DEV02Data Source=PROD-PBIRSDev → Prod
3Initial Catalog=ReportDB_AltInitial Catalog=ReportDBDB-Rename
…beliebig viele weitere Regeln möglich
Optional: Connection String Mapping ist nicht zwingend erforderlich. Ohne aktivierte Regeln werden die Original-Connection-Strings vom Quell-Server 1:1 übernommen. Die Checkbox im Migration-Tab aktiviert / deaktiviert das Mapping.

Die Regeln werden als PowerShell-Hashtable aufgebaut und per -replace-Operator sequentiell auf den Connection-String-Text jeder Datenquelle angewendet. Regex-Syntax ist unterstützt.

Architektur-Prinzip

Alle v3-Funktionen (Invoke-SSRSRequest, Deploy-Report, Deploy-DataSource, Deploy-SharedDataset, New-SSRSFolderRecursive u. a.) bleiben unverändert. Die v4-Neuerungen werden als additive Funktionen und GUI-Erweiterung eingebracht:

  • Get-SSRSBinaryContent — Binär-Download beliebiger Katalog-Items
  • Deploy-PowerBIReport — Multipart-Upload für .pbix
  • Export-SSRSContent — vollständiger Folder-Export inkl. DS-Mapping
